## 背景

项目目前有三个运行入口相互重叠：

- `src/server.ts` 通过底层请求处理器暴露 MCP 工具，并以文本 JSON 返回结果。
- `src/cli.ts` 被打包进 Skill 包，并假设 `session.json` 位于调用方当前工作目录。
- `src/login.ts` 在本地写入 `session.json`，当前构建命令还可能把该凭证复制进 Skill 目录。
- 当前会话模型假设一次只有一个登录用户，即使一个登录用户本身可能已经绑定多个电表。

这个形态适合单个本地仓库和单个南方电网用户，但混淆了源码、构建输出、可携带 Skill 产物、MCP 运行时、私密用户凭证和用户配置选择之间的边界。

```text
变更前形态

src/client 逻辑
   │
   ├── MCP 服务 -> mcp/*.js -> 从多个猜测路径读取会话
   │
   └── Skill 命令行 -> skills/.../scripts/cli.cjs -> 读取 cwd/session.json

构建命令
   ├── 编译代码
   └── 可能把 session.json 复制进 Skill 包
```

## 目标 / 非目标

**目标：**

- 让 MCP 工具结构化、具备结构描述，并兼容当前 MCP SDK 使用方式。
- 让 MCP 产物可零依赖复制部署，部署时不要求携带 `node_modules` 或 `package.json`。
- 从助手视角看，Skill 包应当可携带且行为确定。
- 会话凭证生成和存储默认保持在本地。
- 支持多个本地南方电网用户配置，每个配置有自己的会话、账户列表和电表。
- 支持查询一个用户配置、多个指定电表，或所有已配置用户配置/电表，并报告部分失败。
- 允许助手引导登录，尤其是二维码登录，而不要求用户手动执行每个命令。
- 拆分构建命令，避免普通构建意外打包本地凭证。

**非目标：**

- 除非重构需要，不改变南方电网上游接口协议。
- 不创建托管认证服务。
- 不在远程服务或对话记录中存储凭证。
- 当上游接口要求人工验证、扫码、短信或密码输入时，不保证完全无人值守登录。

## 决策

### 在 MCP 和 CLI 后面使用共享查询服务

创建共享本地模块，用于加载会话、初始化 `CSGClient`、列出账户、查询余额、查询用量和验证会话。MCP 处理器和 Skill 命令行都调用该模块，避免重复结果格式化和账户查找逻辑。

曾考虑保留 MCP 和 CLI 分离并分别更新，但这样会继续重复结果映射和校验逻辑。

### 引入本地用户配置注册表以支持多用户

将“多用户”建模为多个本地用户配置。用户配置是稳定别名，指向一个本地会话文件和相关元数据。MCP 和 Skill 调用时，用户配置别名应当是主要面向用户的标识，而不是原始会话路径。

建议的本地形态：

```text
.csg/
  profiles.json
  sessions/
    home.session.json
    parents.session.json
    shop.session.json
```

`profiles.json` 存储别名、可选显示标签、会话路径、默认用户配置和时间戳。会话文件存储敏感南方电网会话数据。单个用户配置仍然可以包含多个电表，因为上游账户列表可能返回多个用电账户。

部署时，凭证目录优先放在可执行文件同级：

```text
mcp/
  server.cjs
  .csg/
    profiles.json
    sessions/home.session.json

skills/china-southern-power-grid-stat/scripts/
  cli.cjs
  .csg/
    profiles.json
    sessions/home.session.json
```

源码开发运行时仍优先使用项目根目录 `.csg/`，避免把登录凭证写进 `src/.csg/`。已部署产物运行时优先使用运行文件同级 `.csg/`，使复制部署层级更直观。

用户配置选择器支持：

- 未显式选择时使用默认用户配置
- 显式用户配置别名，例如 `home`
- 所有用户配置，用于跨用户汇总

曾考虑让每个命令接收任意会话文件路径数组。它虽然灵活，但会让助手指引更吵、泄露风险更高，也缺少稳定名称供重复使用。

### 将 MCP 迁移为结构化工具结果

尽量使用当前 MCP SDK 的高层工具注册方式。每个工具定义 `inputSchema`、`outputSchema` 和只读标注。成功调用包含 `structuredContent`；`content` 保留紧凑的人类可读摘要或 JSON 文本以兼容旧客户端。

MCP 工具需要感知用户配置：

- 用户配置管理/发现：列出用户配置并验证用户配置会话
- 账户发现：列出一个用户配置或所有用户配置下的账户
- 电表查询：查询所选用户配置下一个户号、多个户号或所有已发现账户的余额或用量

批量结果返回逐用户配置、逐电表条目。部分失败应当体现在结构化结果中，而不是让整个批量结果消失。

曾考虑保留底层请求处理器并手写输出结构。这样可行，但高层接口更能表达意图，也减少协议样板代码。

### 将 MCP 构建为零依赖单文件

MCP 构建输出改为 `mcp/server.cjs` 单文件，使用打包器把 MCP SDK、结构校验依赖和项目查询逻辑一起打入产物。部署时只需要目标环境有 Node.js，不需要安装 npm 依赖，也不需要部署 `package.json`。

MCP 构建清理输出目录时必须保留 `mcp/.csg/`，避免开发者在产物目录旁放置凭证后被普通构建误删。

曾考虑继续输出 `mcp/*.js` 多文件并让部署目录安装生产依赖。这样更接近 TypeScript 默认编译结果，但部署步骤更容易出错，也和 Skill 的零依赖复制体验不一致。

### 让 Skill 会话查找显式且可携带

Skill 命令行按以下顺序查找会话：

1. 显式命令行参数，例如 `--session=/path/to/session.json`。
2. 环境变量，例如 `CSG_SESSION_FILE`。
3. `cli.cjs` 同级 `.csg/` 用户配置目录，或本地专用包中的同级用户配置目录。

命令行不应默认依赖调用方当前工作目录。为了兼容，可以临时保留 `cwd/session.json` 回退路径，但要注明这是旧版兼容行为。

在多用户模式下，Skill 命令行优先使用用户配置选择，而不是原始会话选择：

```text
node ./scripts/cli.cjs --action=profiles
node ./scripts/cli.cjs --action=accounts --profile=home
node ./scripts/cli.cjs --action=accounts --all-profiles
node ./scripts/cli.cjs --action=balance --profile=home --account=...
node ./scripts/cli.cjs --action=balance --profile=home --account=... --account=...
node ./scripts/cli.cjs --action=usage --all-profiles --all-accounts --year=2026 --month=6
```

具体参数可在实现时微调，但契约保持不变：用户配置感知命令在结果中包含用户配置别名，并按用户配置/电表报告部分失败。

曾考虑始终把 `session.json` 嵌入 Skill 包。这样对单机方便，但太容易意外发布或复制凭证。

### 拆分构建和凭证打包

使用独立命令区分开发、MCP 构建、Skill 构建和本地专用 Skill 打包。普通构建只生成代码产物。本地凭证打包命令可以复制会话，但命名和文档必须明确表示仅限本地使用。

建议命令形态：

```text
build              -> 只生成代码产物
build:mcp          -> 生成零依赖 mcp/server.cjs，并保留 mcp/.csg
build:skill        -> 只生成 Skill 脚本
pack:skill-local   -> 可选本地包，将 .csg 复制到 scripts/.csg
login              -> 交互式本地登录
login:agent        -> 助手引导本地登录
```

### 通过编排本地登录支持对话式登录

对话式登录的含义是：助手引导并监控本地登录过程，而不是把凭证存入聊天。登录还应询问会话写入哪个用户配置别名，或是否更新已有用户配置。二维码登录最适合这种流程：

```text
用户要求助手登录
        │
        ▼
助手启动本地登录命令/工具
        │
        ▼
系统生成二维码登录链接
        │
        ▼
助手展示链接/二维码并轮询状态
        │
        ▼
上游确认扫码
        │
        ▼
会话在本地验证并写入
用户配置注册表创建/更新
```

短信验证码登录也可以对话式完成：助手请求手机号、触发短信发送，并请用户提供验证码。密码登录应使用本地隐藏终端提示或等价安全输入。指引应避免鼓励用户在对话中发送密码。

实现选项：

- CLI 优先：增加支持二维码/短信流程并输出机器可读状态事件的登录命令模式。助手运行命令并转述进度。
- MCP 优先：增加 `start_login`、`check_login_status`、`finalize_login` 等登录 MCP 工具。集成度更高，但会触及 MCP 服务的安全模型。

初期优先 CLI 方案，因为它让认证保持本地，也避免把长时间运行的登录状态机放进查询 MCP 服务。

## 风险 / 取舍

- 凭证通过打包泄露 -> 通过凭证无关的普通构建和命名明确的本地凭证打包命令缓解。
- 构建清理误删部署凭证 -> MCP 构建只清理代码产物并保留 `mcp/.csg/`，Skill 本地凭证打包只在显式命令中写入 `scripts/.csg/`。
- 助手混淆会话路径 -> 通过确定性查找顺序和解释配置方式的 JSON 错误缓解。
- 助手混淆用户/电表 -> 通过命名用户配置、列出用户配置命令/工具、带用户配置标记的账户结果，以及无默认用户配置时的明确错误缓解。
- 多用户部分失败 -> 通过逐用户配置/逐电表结果条目缓解，避免全有或全无。
- MCP 客户端期望纯文本结果 -> 同时返回 `structuredContent` 和兼容 `content`。
- 密码暴露在对话中 -> 优先二维码登录，密码使用本地隐藏输入。
- 二维码轮询复杂 -> 使用有界超时、取消行为和清晰重试输出。
- 重构导致生成产物变动较大 -> 先改源码，再重新生成 `mcp/` 和 Skill 脚本输出。

## 迁移计划

1. 增加用户配置注册表、共享查询/会话辅助函数和用户配置选择器解析。
2. 更新 MCP 和 CLI，使二者使用共享的用户配置感知查询层。
3. 更新 MCP 工具为结构化输入、输出和结果。
4. 更新 Skill 命令行的会话查找、用户配置参数、批量查询行为和错误输出。
5. 拆分代码构建和本地凭证打包命令。
6. 更新 `SKILL.md` 和 `README.md`，反映新的运行时、用户配置和凭证规则。
7. 增加或更新二维码/短信助手引导登录，以及用户配置创建/更新行为。
8. 重新生成零依赖 MCP 单文件和 Skill 产物。

回滚在本地项目中比较直接：在新的命令行和 MCP 服务通过本地验证前，保留旧的 `mcp/` 与 Skill 脚本输出即可。
